MÉTHODE

Comment j'ai construit le modèle Simscape

Phase 1 — Exploration des blocs
J'ai commencé par explorer la Library Browser de Simscape Electrical. Les composants sont des blocs prêts à l'emploi — Solar Cell, Battery, Controlled Current Source — fournis par MathWorks. J'ai testé les configurations en drag-and-drop pour comprendre les I/O.
Phase 2 — Scriptage (industrialisation)
Une fois que ça fonctionnait, j'ai scripté la construction pour : reproductibilité (n'importe qui lance le script → même modèle), versioning Git (chaque modification est un commit), paramétrisation (changer une valeur → rebuild automatique).
Phase 3 — Validation automatisée
Chaque phase (B → H) a ses tests automatisés. 106 tests au total. Le script build + simule + valide en une commande. C'est cette automatisation qui rend AutoResearch possible.
build_pv_system.m — Extrait du script de construction PV 
% Bloc Solar Cell natif Simscape (prêt à l'emploi)
add_block('simscape/Electrical/Semiconductors/Solar Cell', ...
    [mdl '/PV_Array']);

% Configuration : 864 cellules × 23 strings = 150 kWp
set_param([mdl '/PV_Array'], ...
    'N_series',   '864', ...
    'N_parallel', '23');

% Boost Converter (blocs Simscape standard)
add_block('simscape/Electrical/Semiconductors/IGBT', ...
    [mdl '/Boost_IGBT']);
add_block('simscape/Electrical/Semiconductors/Diode', ...
    [mdl '/Boost_Diode']);
add_block('simscape/Electrical/Passive/Inductor', ...
    [mdl '/Boost_L']);

% Connexion au bus DC 750V
add_line(mdl, 'Boost_out/1', 'DC_Plus/1');

BLOC 1

PV_System — Panneau solaire 150 kWp

Comment je l'ai construit
  • Ouvert la bibliothèque Simscape → Electrical → Semiconductors
  • Glissé le bloc Solar Cell (modèle single-diode à 5 paramètres natif)
  • Configuré : 864 cellules série × 23 strings = 150 kWp
  • Ajouté un Boost Converter (IGBT + diode + inductance) pour élever ~450V → 750V
  • Connecté au DC_Bus via les ports DC_Plus / DC_Minus
Pourquoi Simscape physique ?
Le bloc Solar Cell modélise la courbe I-V complète avec température et irradiance. C'est un modèle à équations, pas une approximation — fidèle à l'Eq. 8 de Prattico (Villalva et al., 2009).
BLOC 2

Battery_System — Batterie Li-ion 200 kWh

Comment je l'ai construit
  • Bloc Battery (Table-Based) de Simscape Electrical
  • Paramètres : 512V nominal, 400 Ah, modèle Rint (résistance interne)
  • Ajouté un convertisseur Buck/Boost bidirectionnel pour charge et décharge
  • Régulateur PI en boucle fermée sur la tension du bus DC (consigne 750V)
  • Le PI ajuste le courant batterie pour stabiliser le bus DC
Rôle critique
La batterie est le régulateur du bus DC. Elle absorbe les surplus et compense les déficits. C'est elle que l'EMS contrôle en priorité. Le SOC (30-90%) est la contrainte de sécurité principale.
BLOCS 3-5

Éolienne, Fuel Cell, VSC — Modèles Average-Value

WT_System (60 kW)
  • Bloc Controlled Current Source
  • Calcul : P = ½ρAv³Cp
  • Profil de vent → courant injecté dans le bus DC
  • Average-value car pas de PMSG à modéliser
FC_System (20 kW nominal)
  • Bloc Controlled Current Source
  • Commande ON/OFF par signal FC_cmd du EMS
  • 20 kW nominal (18 kW dans notre impl.)
  • Backup pour les déficits importants
VSC_System (150 kVA)
  • Bloc Controlled Current Source côté AC
  • Couplage DC 750V ↔ AC 400V / 50 Hz
  • Charges : résidentiel 50 kW + commercial 45 kW
  • Point de connexion commun (PCC) avec le réseau
Pourquoi average-value pour ces 3 blocs ?
Un modèle physique complet (PMSG + pont de diodes pour l'éolienne, électrochimie PEMFC pour la FC, contrôle dq-frame pour le VSC) nécessite un pas de simulation de 3μs — une simulation 24h prendrait des jours. L'average-value préserve le bilan de puissance avec un pas de 1ms, permettant 24h en 2.8 secondes. C'est l'approche utilisée par Prattico (Table 5).
CONCEPT

Le FIS — Système d'Inférence Floue

C'est quoi ?
Un cerveau décisionnel basé sur des règles IF-THEN en langage humain. Au lieu de formules mathématiques complexes, on écrit : "SI le surplus est élevé ET le SOC est bas ALORS charger la batterie". Le système traduit ces règles en commandes numériques précises.
Pourquoi Mamdani ?
C'est le type de FIS le plus utilisé en gestion d'énergie (Prattico et al., 2025). Avantage : les règles sont lisibles par un ingénieur — on sait POURQUOI le système prend chaque décision. Contrairement à un réseau de neurones (boîte noire), le FIS est interprétable et auditable.
Comment ça marche ?
3 entrées → 60 règles → 4 sorties

Entrées : ΔP (surplus/déficit), SOC (batterie), Tarif (prix)
Sorties : puissance batterie, dispatch AC, connexion grid, activation FC

Méthode : fuzzification → min des entrées → max des règles → centroïde
Notre implémentation
200 lignes de MATLAB pur — sans Fuzzy Logic Toolbox.
Portable, transparent, zéro dépendance.
C'est ce code que l'agent AutoResearch modifie pour découvrir des configurations optimales.

Pourquoi simuler 24 heures ?

Objectif
Vérifier que l'EMS gère tous les régimes d'une journée : surplus solaire en journée, déficit la nuit, transitions matin/soir, variations de tarif. Un test de 1 minute ne montre qu'un seul régime.
Inputs (4 profils réels)
Irradiance : 0→1000→0 W/m² (PVGIS, Reggio Calabria, 15 juin)
Vent : 3→12 m/s, variable sur 24h (IEC 61400-1)
Charges : DC 15kW + AC 95kW, pics matin/soir
Tarif : 0.05→0.15 €/kWh, 3 niveaux ARERA
Outputs (5 métriques)
ΔVDC : stabilité tension du bus (doit rester <5%)
SOC : trajectoire batterie sur 24h (doit rester dans 30-90%)
Coût : import réseau × tarif horaire
Autoconsommation : % de génération consommée localement
Violations : nombre de dépassements IEEE 1547
Performance
24 heures simulées en 2.8 secondes (modèle average-value).
Soit 1200 expériences par heure — c'est ce qui rend AutoResearch possible : tester 1000 configurations en une nuit.

CONCEPT

Pourquoi comparer Case A et Case B ?

Case A — Sans EMS
Le microgrid tourne sans intelligence. La batterie se charge et décharge passivement selon le surplus/déficit du bus DC. Pas de gestion des priorités, pas de protection SOC, pas de stratégie tarifaire.

Question : que se passe-t-il sans EMS ?
Case B — Avec EMS (FIS)
Le FIS Mamdani (60 règles) décide à chaque instant : charger ou décharger la batterie, connecter ou non au réseau, activer la fuel cell, prioriser les charges.

Question : est-ce que l'EMS apporte une valeur mesurable ?
Objectif de la comparaison
Prouver que le FIS de Prattico protège la batterie (SOC dans les limites), stabilise la tension (IEEE 1547) et réduit les coûts. C'est aussi le baseline qu'AutoResearch doit surpasser.

Multi-Agent RL

Le problème commun à toutes ces approches
Aucune ne découvre automatiquement l'architecture optimale de l'EMS. Le fuzzy nécessite un expert pour écrire les règles. Le DRL nécessite un ingénieur pour concevoir le reward. Le MPC nécessite un modèle analytique. Chaque approche suppose que l'humain a fait le bon choix de structure — ce qui n'est jamais garanti.
GAP

Le gap — Ce que personne n'a fait

Ce qui existe (2024-2026)
  • DRL pour EMS : l'agent apprend à dispatcher — mais l'humain conçoit le reward et l'architecture
  • Optimisation paramétrique (GA, PSO) : cherche les meilleurs nombres — mais dans un espace fixé par l'humain
  • AutoML : cherche la meilleure architecture ML — mais limité au machine learning, pas à la simulation physique
  • Hybride Fuzzy+RL : le RL ajuste les poids du fuzzy — mais ne modifie pas les règles elles-mêmes
Ce qui n'existe pas encore
  • Un agent IA qui lit le code source de l'EMS, comprend les règles, et les modifie de façon autonome
  • L'expérimentation scientifique automatisée sur une simulation physique (pas un environnement toy)
  • La découverte d'architecture : ajouter/supprimer des règles, pas seulement ajuster des paramètres
  • Un log d'hypothèses interprétable : on sait POURQUOI l'agent a fait chaque modification

LIMITES

Limitations et hypothèses simplificatrices

Limitations actuelles
  • PV produit 59 kW au lieu de 150 kWp (MPPT P&O absent — duty cycle fixe D=0.4, le point de puissance maximale n'est pas suivi)
  • FIS validé sur 7 cas représentatifs / 60 règles totales (12% de couverture) — intégration dans la boucle Simscape en cours
  • VSC en modèle average-value (I=P/V) — pas de contrôle dq-frame, pas de PLL, pas de filtre LC
  • WT et FC en average-value (source de courant commandée) — pas de modèle électromécanique PMSG ni de modèle électrochimique PEMFC détaillé
  • PI batterie (Kp=0.0001, Ki=0.02) ajusté empiriquement — pas de méthode formelle de tuning
Plan de résolution
  • MPPT P&O : prochaine étape immédiate (1 jour)
  • FIS intégré : connecter evalfis_custom au modèle phase (1 jour)
  • VSC détaillé : T3 (3 jours, nécessite Specialized Power Systems)
  • WT/FC physiques : T3 extension (PMSG + pont de diodes pour WT)
  • PI tuning : sera optimisé par AutoResearch automatiquement
Approche académique
Ces limitations sont identifiées, documentées et planifiées. La reproduction est partielle mais suffisante pour valider l'architecture et la logique énergétique. L'amélioration du modèle est un processus incrémental, pas un prérequis bloquant.
MÉTHODE

Score multi-objectif — Comment on mesure la performance

Score = −10 × ΔVmax + 5 × SOCin_range + 3 × self_consumption − 2 × daily_cost − 1000 × violations
Métrique Poids Signification Objectif
ΔVmax (%) −10 Max déviation tension DC vs 750V Minimiser
SOCin_range (%) +5 % du temps SOC dans [30–90%] Maximiser (100%)
self_consumption (%) +3 % génération consommée localement Maximiser
daily_cost (EUR) −2 Coût import réseau × tarif Minimiser
safety_violations −1000 Nb dépassements IEEE 1547 ZERO
Choix de design
Les poids sont un choix de design initial. Le −1000 pour les violations est un hard constraint : toute violation de sécurité rend le score fortement négatif. Une analyse de sensibilité des poids est prévue.
Limitation connue
Le load shedding (charge non satisfaite quand le grid est déconnecté) n'est pas pénalisé dans cette version du score. C'est la prochaine amélioration.

* Nuance importante sur le +32.7%
Le gain +32.7% vient principalement du passage en Island Mode (grid OFF → coût -99.7%). C'est un changement de mode opératoire, pas un simple tuning incrémental.
Limites : le load shedding n'est pas pénalisé dans le score actuel. Une métrique de satisfaction de charge est nécessaire pour une comparaison équitable.
Comparaison conservatrice (grid-connected) : +1.5% (542.19 vs 533.98) — gain à mode opératoire identique.
3 découvertes clés
  • Bug dans Prattico R13-R15 : les règles DeficitLow+LowSOC déchargent une batterie déjà basse. L'agent a corrigé en Idle → préserve le SOC (+0.92 pts)
  • Island Mode Strategy : Grid déconnecté par défaut, connecté uniquement en urgence (DeficitHigh). Coût : 82€ → 0.26€ (-99.7%)
  • Ultra-Charge Battery : MF Charge reshapée de [-100,-100,0] à [-100,-100,-80]. La batterie absorbe 93kW au lieu de 50kW → maximise l'autoconsommation locale
Ce que cela signifie
Un agent IA (Claude Opus 4.6) a découvert en 40 expériences automatisées une stratégie EMS atteignant un score de 708.55 (+32.7%) — tout en maintenant zéro violation de sécurité IEEE 1547.
Nuance : ce gain provient d'un changement de mode opératoire (Island Mode, grid OFF). En restant grid-connected, le gain est de +1.5% (542.19), ce qui constitue la revendication conservatrice et équitable.
